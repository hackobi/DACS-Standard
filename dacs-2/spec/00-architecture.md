# DACS/2 §0 — Architecture

## 1. Layers

DACS/2 is specified in two layers. Conformance to L0 alone is portable; interoperability requires L1.

- **L0 — Artifact layer.** Defines canonical form, content addressing, domain-separated signatures, and the schemas of
  every lifecycle artifact (listing, agreement, evidence, bundle, …). An L0 verifier can validate any DACS/2 artifact
  offline given its bytes and the signer's keys. L0 has no Demos dependency.
- **L1 — Demos binding.** Defines where artifacts live and how parties talk: which Demos transaction anchors which
  artifact, which RPC reads resolve it, which transport carries envelopes, and which settlement rails move value.
  The complete binding table is [bindings/demos-node.md](../bindings/demos-node.md). L1 is normative for the
  `dacs2-demos` conformance class.

*Rationale:* v0.1 specified five abstract substrate requirements and left two of them without wire formats. DACS/2
inverts this: the portable part is exactly the artifact layer (which truly is portable), and everything that requires a
substrate is bound to one concrete, open-source substrate with exact names. A future third edition MAY add bindings for
other substrates; until one exists, claiming substrate neutrality would be false.

## 2. Actors

| Actor | Definition |
|---|---|
| **Buyer agent** | Demos account (ed25519 root) acquiring a deliverable. Executes pay-* phases. |
| **Seller agent** | Demos account offering a listing. Executes deliver-* phases. |
| **Steward group** | N accounts in the `stewards` ACL group of the registry Storage Program; co-sign recipes and rails (M-of-N, see [spec/03 §6](03-vetting.md)). |
| **Arbitrator** | A seller of dispute-resolution services, credentialed via DACS-1/2, selectable in DACS-6. |
| **Validators** | Demos PoRBFT validator shard — stake-weighted (`validatorStake`, min 10^18 OS) with deterministic shard selection over active validators. Provide anchoring finality, DAHR execution, and TLSNotary infrastructure. Not DACS-aware. |
| **Catalog mirror** | Optional off-chain indexer of listing programs. Non-authoritative; clients MUST verify against on-chain state. |

There is **no orchestrator role**. v0.1's orchestrator invoked phases, classified errors, ran auctions, and assembled
bundles, while being unspecified, unaccountable, and reputation-exempt. In DACS/2:

- Each session phase has a single **executor** determined by phase kind (§4); the executor signs that phase's artifacts.
- Sealed-envelope selection is deterministic from anchored commits and reveals (spec/05 §5); no trusted selector exists.
- Each party assembles and anchors **its own** bundle copy (spec/07 §4).
- A party MAY delegate execution to a facilitator, but the facilitator acts under the party's session key
  (spec/02 §5) and the party remains the accountable principal. Facilitation is itself a DACS service with a listing
  and a reputation.

## 3. Identifiers

- **Account**: Demos ed25519 public key, `0x` + 64 lowercase hex. The root of all identity (spec/02).
- **`jobId`**: ULID, generated by the buyer at session start, unique per session. Every artifact and every session-bound
  settlement transaction carries it.
- **`listingId`**: seller-chosen, ≤ 128 URL-safe ASCII, unique per seller account.
- **`channelId`**: `sha256hex(jobId || ":" || buyerAccount || ":" || sellerAccount)` — deterministic, collision-free per session.

## 4. Session model (peer-driven)

A **session** is the execution of a listing's declared `pipeline` between exactly one buyer and one seller.

1. Buyer resolves the listing program, verifies hash + signature, and opens a channel (`DacsEnvelope` transport, spec/05 §2).
2. Stages run in pipeline order. Per-phase executor:

   | Phase kind | Executor | Counterparty obligation |
   |---|---|---|
   | `vet-credentials` | The party imposing the requirement (each side vets the other) | Present bundle on request |
   | `negotiate-*` | Both (symmetric message exchange) | — |
   | `commit-agreement` | Buyer anchors; both co-sign | Co-sign within `commitTimeoutSec` |
   | `pay-*` | Buyer (payer) | Acknowledge via envelope |
   | `deliver-*` | Seller | Acknowledge / dispute via envelope |
   | `rate` | Either, optional | — |
   | `finalise` | Both: each anchors its own bundle | Co-sign counterparty bundle hash |

3. The session state machine (spec/07 §2) is evaluated independently by each party from the same anchored facts;
   divergence is detectable from the two bundle copies and is the entry condition of DACS-6.

A phase executor reports `PhaseResult {ok, reason?, errorClass?, txRefs?, attestationRef?}` with
`errorClass ∈ {permanent, transient, counterparty, substrate, settlement-atomicity}` — semantics unchanged from v0.1,
but classification by an executor is only reputation-effective when counter-signed (spec/07 §5).

## 5. Trust model summary

- **Artifact integrity**: Ed25519/Falcon/ML-DSA signatures over domain-separated canonical hashes (spec/01).
- **Anchoring**: Demos PoRBFT finality (2/3+1 shard vote, ~10 s blocks). An anchored artifact is non-repudiable and
  totally ordered relative to other anchors.
- **Identity**: chain-native claims are GCR state verified by any node read; authority claims inherit the trust of
  their DACS-2 method, with tier floors (spec/03 §4).
- **Counterparty risk**: bounded by rail choice — escrow and conditional-release rails remove the
  pay-then-pray asymmetry; plain rails retain it and listings MUST disclose the ordering (spec/06).
- **Out of scope** (unchanged from v0.1, stated plainly): collusion prevention, network-level DoS, regulatory
  enforcement, unconditional privacy.

## 6. Conformance classes

| Class | Requirements |
|---|---|
| `dacs2-core` | L0: canonicalisation, signatures, schema-valid artifacts. MUST pass schema validation for every artifact kind it emits. |
| `dacs2-demos` | L0 + L1 in full: all six stages over a Demos network. |
| `dacs2-demos/<stage>` | L0 + the L1 binding for one stage. MUST declare the profile list it implements in its agent card (spec/04 §5). |

Partial implementations MUST NOT emit artifacts for stages they do not implement.

## 7. Versioning

- Artifacts carry `dacs: "2"` and use `:v2:` separators. Major versions break; within a major, unknown fields MUST be
  preserved when hashing (SIG-5) and MUST NOT cause rejection.
- The Demos binding is versioned independently (`binding: "demos/1"`) and declares per-feature gates rather than a
  single version switch, because Demos networks upgrade asynchronously ([bindings/demos-node.md](../bindings/demos-node.md) §6).
- Registries (schemes, methods, rails, separators) are closed; additions require a steward-group entry in the on-chain
  registry program (spec/03 §6). `x-` prefixed kinds remain the experimental escape valve and MUST NOT appear in
  reputation inputs.
